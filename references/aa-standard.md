# AA 分层规范 (Application Architecture)

应用架构描述为实现业务能力所需的应用系统与服务组件。

---

## L1 产品

IT 战略版图，主权边界与 DA-L1 主题域分组严格对齐。

- **对齐**：AA-L1 主权边界必须与 DA-L1 一致
- **命名**：`[业务领域名称] + [系统/平台]`
- **示例**：客户关系管理系统（CRM）、客户成功管理平台（CSM）、产品与交付平台

---

## L2 子产品

核心对象的"管理中心"，与 DA-L2 主题域对齐。

- **命名**：`[核心对象] + [管理中心/子系统]`
- **示例**：商机管理中心、客户档案管理中心、续约管理中心、客户成功与健康中心

---

## L3 应用/微服务

业务场景运行单元，逻辑上对应一个限界上下文（Bounded Context）。

- **原则**：
  - **逻辑独立**：每个 L3 应用封装一个限界上下文，可独立演化
  - **伸缩缝原则**：逻辑拆分，物理部署可合并（根据规模决定）
  - 与 DA-L3 主题子域对齐
- **命名**：`[场景/范畴修饰词] + [应用/服务/引擎]`
- **示例**：客户画像应用、健康评分引擎、风险预警引擎、入网计划编排服务、续约管理应用

> **伸缩缝原则**：L3 应用在逻辑上相互独立，可分别迭代；物理部署时可因团队规模或流量需求合并或拆分，不影响逻辑边界。

---

## L4 服务/模块

> **N:1:1 推导的 1 端（应用侧）— 与 DA-L4 BO 严格 1:1 对应。**

每一个 DA-L4 BO 有且仅有一个 AA-L4 服务作为其唯一逻辑代理。

- **原则**：高内聚，封装单个 BO 的全部状态机与业务规则；作为该 BO 在系统中的唯一权威管理者
- **命名**：`[DA-L4 BO 名称] + [服务]`
- **示例**：报价单服务、健康评分服务、续约合同服务、成功计划服务、风险预警服务

**1:1 约束的意义：**

| 违规模式 | 问题 |
|---------|------|
| 1 个服务管理多个 BO | → 逻辑烟囱，状态机交叉，难以拆分 |
| 1 个 BO 被多个服务管理 | → 状态机分裂，数据一致性风险 |
| 服务命名与 BO 不一致 | → 语义歧义，蓝图与代码不对应 |

---

## L5 功能/接口

逻辑实体的"原子操作行为"，直接指导 API 设计。

- **命名**：`[动作] + [实体名] + [API]`
- **常见动作**：创建、查询、更新、删除、提交、审批、发布、归档、关闭
- **示例**：
  - 创建解决方案提案 API
  - 查询客户健康评分 API
  - 提交续约合同审批 API
  - 发布成功计划 API
  - 关闭风险预警 API
